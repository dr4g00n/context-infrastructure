# 数据安全核心理念

## 核心原则

数据安全不是产品的附加功能，而是系统设计的基石。以下是我对数据安全的核心认知：

### 1. 数据全生命周期视角

数据安全必须覆盖数据从产生到销毁的完整生命周期：

```
创建 → 存储 → 使用 → 传输 → 归档 → 销毁
```
每个阶段都有独特的风险和防护需求。

### 2. 默认不信任

- **零信任假设**：网络内部不再默认安全
- **最小权限原则**：仅授予必要的最小访问权限
- **纵深防御**：多层控制，单点失效不等于系统失效

### 3. 安全左移

- 在设计阶段而非部署后考虑安全
- [[威胁建模]] 应成为标准流程
- [[安全代码审查]] 早期介入

## CIA 三要素（核心）

### 机密性 (Confidentiality)
确保数据仅被授权方访问

- [[加密技术]]：静态加密 + 传输加密
- [[访问控制]]：基于角色/属性的权限管理
- [[数据脱敏]]：生产数据用于开发/测试时的处理

### 完整性 (Integrity)
确保数据未被未授权篡改

- [[哈希校验]]：检测数据是否被修改
- [[数字签名]]：验证数据来源和完整性
- [[审计日志]]：记录所有变更操作

### 可用性 (Availability)
确保授权用户在需要时可以访问数据

- [[备份策略]]：定期备份 + 灾难恢复
- [[高可用架构]]：消除单点故障
- [[DDoS 防护]]：保障服务连续性

## 设计决策框架

当面临安全权衡时，优先级排序：

1. **用户数据安全** > 系统可用性
2. **不可逆损害防护** > 可恢复操作
3. **自动化防御** > 人工响应
4. **透明告知** > 隐瞒处理

## 常见误区

| 误区 | 正确认知 |
|------|---------|
| "我们已经加密了，所以安全" | 加密不是银弹，密钥管理、访问控制同样重要 |
| "内网是安全的" | 威胁可能来自内部，零信任是趋势 |
| "合规即安全" | 合规是底线，不是上限 |
| "安全阻碍业务" | 好的安全是业务的赋能者 |
