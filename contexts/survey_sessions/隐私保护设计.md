# 隐私保护设计

## 核心原则

隐私保护不是数据收集的阻碍，而是产品设计的基石。

**"隐私不是事后补救，而是设计起点"**

---

## Privacy by Design (PbD) 七大原则

### 1. 主动而非被动
- 默认隐私保护
- 预防而非补救

### 2. 隐私是默认设置
- 自动保护个人数据
- 用户无需手动开启

### 3. 隐私嵌入设计
- 融入产品架构
- 不是附加组件

### 4. 全功能设计
- 隐私不影响功能
- 两者可以共存

### 5. 端到端安全
- 全生命周期保护
- 从收集到销毁

### 6. 可见性与透明度
- 用户知情
- 处理过程可见

### 7. 尊重用户隐私
- 用户主导
- 以用户为中心

---

## 数据最小化原则

| 原则 | 说明 |
|------|------|
| **必要收集** | 仅收集业务必需的数据 |
| **一次性使用** | 数据用完即删除，不长期保留 |
| **去标识化** | 收集时即做去标识处理 |
| **匿名化** | 尽可能使用匿名数据 |

## 隐私保护技术

### 技术层
- [[数据脱敏]]：脱敏、掩码、加密
- 差分隐私：添加噪声保护个体
- 联邦学习：数据不出本地
- 同态加密：加密状态下计算
- 安全多方计算 (MPC)：多方联合计算

### 流程层
- 隐私影响评估 (PIA)
- 数据保护影响评估 (DPIA)
- [[审计日志]]：访问记录
- [[事件响应流程]]：泄露响应

## 合规框架

| 框架 | 地域 | 核心要求 |
|------|------|---------|
| GDPR | 欧盟 | 明确同意、被遗忘权、数据可携 |
| CCPA | 美国加州 | 知情权、删除权、选择退出 |
| PIPL | 中国 | 同意、最小必要、本地化存储 |
| PDPA | 新加坡 | 同意、目的限制、保留期限 |

## 设计检查清单

**数据收集阶段：**
- [ ] 是否明确告知用户收集目的？
- [ ] 是否获得用户明确同意？
- [ ] 是否仅收集必要数据？
- [ ] 默认隐私设置是否安全？

**数据使用阶段：**
- [ ] 是否按声明的目的使用？
- [ ] 是否有访问控制？
- [ ] 敏感数据是否脱敏？
- [ ] 是否记录使用日志？

**数据存储阶段：**
- [ ] 是否加密存储？
- [ ] 是否有保留期限？
- [ ] 是否有备份策略？
- [ ] 是否有泄露监控？

**数据销毁阶段：**
- [ ] 是否按期删除？
- [ ] 是否彻底清除？
- [ ] 是否有删除审计？
