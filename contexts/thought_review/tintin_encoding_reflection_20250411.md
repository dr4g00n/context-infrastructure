# 反思报告：tintin 项目编码问题调试

**日期**: 2026-04-11  
**项目**: 书剑 MUD / tintin++ 智能寻路系统  
**主题**: 为什么第一时间没想到 `#config charset GBK1TOUTF8`？

---

## 1. 问题回顾

### 1.1 遇到的错误

```
#ERROR: UNKNOWN TINTIN COMMAND '书剑'
#ERROR: UNKNOWN TINTIN COMMAND '使用方法:'
#ERROR: UNKNOWN TINTIN COMMAND '涔﹀墤'
#ERROR: UNKNOWN TINTIN COMMAND '浣跨敤鏂规硶'
```

### 1.2 根因

`start.sh` 的处理流程有缺陷：

```bash
# 错误流程
iconv -f UTF-8 -t GB18030 main.tt > main_gb.tt   # 步骤A
sed -i.bak 's/foo/bar/' main_gb.tt                # 步骤B（破坏发生在这里）
```

`sed` 不知道 `main_gb.tt` 是 GB18030 编码，按 UTF-8 解析字节流，导致多字节汉字被截断。

---

## 2. 决策路径分析

### 2.1 当时实际看到的信息

`main.tt` 第11行：

```bash
#config charset GBK1TOUTF8
```

**状态**: 被注释掉了，伴随着一大段被注释的模块加载代码。

### 2.2 实际发生的思维链条

| 步骤 | 观察 | 推导 |
|------|------|------|
| 1 | 所有 `#` 开头的行都被注释 | `#config` 也是注释状态，不是"激活配置" |
| 2 | 存在 `start.sh` 做 iconv 转换 | "项目原本的设计就是用 iconv 预处理" |
| 3 | 没有质疑 iconv 方案的合理性 | "既然有 start.sh，那就按这个路径修 bug" |
| 4 | 陷入 sed 编码问题的细节 | 精力被具体错误吸引，没有抬头看全局 |

### 2.3 关键盲点

**框架惯性**：被既有的工程结构"带偏"了。

```
看到 start.sh 存在 → 假设这是正确方案 → 在错误方向上努力
        ↑___________________________________________|
                    （没有质疑前提）
```

**更好的第一问题**应该是：

> "为什么要用 iconv 预处理？tt++ 本身不支持 UTF-8 吗？"

而不是：

> "iconv + sed 为什么会破坏编码？"

---

## 3. 根本原因

### 3.1 对"被注释代码"的误读

被注释的 `#config charset GBK1TOUTF8` 不只是"没激活的配置"，它是**前人留下的线索**：

- 可能试过但有问题
- 可能是更好的方案但当时没采用
- 可能是遗留代码等待重新启用

**正确态度**：看到被注释的配置，要好奇"为什么被注释"。

### 3.2 工具链迷信

`start.sh` 存在 ≠ 它是最佳实践  
`start.sh` 存在 = 它只是"能工作的第一个版本"

### 3.3 复杂性问题

| 方案 | 复杂度 | 问题 |
|------|--------|------|
| iconv → sed → tt++ | 高 | 编码链条长，容易断裂 |
| tt++ 内置 charset | 低 | 单点配置，tt++ 内部处理 |

**奥卡姆剃刀原则**：能用内置功能解决，就不用外部链条。

---

## 4. 可复用的教训

### 教训 A：质疑既有方案的前提

当修复一个复杂 bug 时，停下来问自己：

> "这个问题本身是不是不应该存在？有没有更直接的解决路径？"

### 教训 B：探索顺序优化

**正确的探索顺序**：

```
1. 看 main.tt 顶部的注释
   └── 发现 #config charset GBK1TOUTF8 被注释

2. 查 tt++ 手册 / 搜索
   └── 确认 tt++ 原生支持 GBK1TOUTF8 转码

3. 测试
   └── 取消注释，直接 tt++ main.tt
       └── 成功！无需 iconv 链条

4. 结论
   └── 删除 start.sh 的复杂逻辑，简化流程
```

### 教训 C：区分"存在"与"正确"

| 观察 | 推论 | 验证 |
|------|------|------|
| start.sh 存在 | 这是项目设计的方案 | ❌ 未验证 |
| #config 被注释 | 这个配置不可用 | ❌ 未验证 |

**正确做法**：每个观察都要独立验证，不能被工程结构带偏。

## 6. 关联概念

### GB18030 编码陷阱

GB18030 是变长编码：

| 字符类型 | 字节数 | 示例 |
|---------|-------|------|
| ASCII | 1字节 | `A` = `0x41` |
| 常用汉字 | 2字节 | `中` = `0xD6 0xD0` |
| 生僻汉字 | 4字节 | 扩展字符 |

**破坏过程**：

```
UTF-8  "书剑" = E4 B9 A6 E5 89 91 (6字节)
    ↓ iconv
GB18030 "书剑" = CA E9 BD A3 (4字节)
    ↓ sed (按 UTF-8 解析)
解析结果：CA E9 → 无效 UTF-8 序列 → 乱码
```

### tt++ charset 配置

```bash
#config charset GBK1TOUTF8
```

- 发送时：UTF-8 → GBK
- 接收时：GBK → UTF-8
- 内部处理：tt++ 自己完成，无需外部干预
