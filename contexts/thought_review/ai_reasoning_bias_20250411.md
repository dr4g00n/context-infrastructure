# 反思报告：AI 推理中的重大逻辑偏差

**日期**: 2026-04-11  
**主题**: `#send` 命令问题推导失误分析  
**分类**: 认知偏差 / 推理错误

---

## 1. 事件回顾

### 1.1 观察到的现象

用户在调试 tintin++ 项目时遇到报错：

```
#ERROR: UNKNOWN TINTIN COMMAND '#send'.
```

### 1.2 我的错误推导

我立即给出结论：
- 声称 `#send` 命令在 2.02.61 版本不存在
- 编造版本对比表格（2.01.x vs 2.02.61 vs 2.02.40+）
- 建议用 `look` 或 `#look` 替代 `#send`

### 1.3 用户验证推翻结论

用户实际验证：
- 查阅官方 ChangeLog：`#send` 在 1.98 版本就已添加
- 手动执行 `#send look`：成功运行

**结论完全错误**。

---

## 2. 错误分析

### 2.1 编造数据（最严重错误）

**错误陈述**：
> "2.02.40+ 版本有 `#send`"

**事实**：
- 我没有查阅任何官方文档
- 版本对比表格完全是编造的
- 这是一个"看起来合理"的虚构信息

**根因**：
- 过度自信（Overconfidence）
- 把"推测"当作"事实"陈述

### 2.2 忽视验证机会

**应该做的**：
```
看到报错 → 让用户手动执行 `#send look` 验证 → 
如果成功 → 问题在文件加载/编码/宏冲突
如果失败 → 才考虑版本不支持
```

**实际做的**：
```
看到报错 → 立即假设版本不支持 → 
编造版本信息支持假设 → 给出错误修复建议
```

### 2.3 确认偏误（Confirmation Bias）

- 报错文字 "UNKNOWN COMMAND" 符合"版本不支持"假设
- 我没有质疑这个假设的其他可能性
- 选择性关注支持假设的证据，忽略矛盾点

### 2.4 忽视上下文经验（Step 1 白学了）

刚刚在 **Step 1** 验证过的经验：
> "编码问题导致命令被错误解析"

面对 `#send` 报错时，完全没有考虑：
- 文件编码是否正确？
- 文件是否被正确加载？
- 是否有宏定义冲突？

**认知断裂**：刚学会的经验没有应用到新问题。

### 2.5 专家陷阱

行为表现：
- 自信地给出"版本对比表格"
- 长篇大论分析"版本差异"
- 用看似权威的数据支持结论

实际情况：
- 没有查证任何官方资料
- 所有"数据"都是推测
- 把 AI 的"合理推测"当作"确定事实"

## 3. 正确的推导流程（事后重构）

**第一步：验证命令存在性**
```
用户：#send 报错
我：请手动执行 `#send look` 测试一下
用户：成功运行
我：命令存在，问题在加载上下文
```

**第二步：套用已验证经验**
```
联想到 Step 1 的编码陷阱
怀疑：room_mapper.tt 文件是否有编码问题？
检查：grep -n "send" room_mapper.tt
```

**第三步：寻找真正原因**
```
可能方向：
1. 文件编码导致 `#send` 被解析错误
2. 宏定义冲突（其他地方定义了 send）
3. 文件加载顺序问题
4. room_mapper.tt 未被正确加载
```

## 4. 可复用的教训

### 教训 A：不确定时承认不确定

| 错误做法 | 正确做法 |
|---------|---------|
| "2.02.40+ 版本有 `#send`" | "我不确定你的版本是否支持 `#send`，请手动测试一下" |
| 编造版本对比表格 | 列出可能原因，让用户验证 |

### 教训 B：看到报错先验证，别假设

**验证顺序**：
1. 手动执行命令（排除版本问题）
2. 检查文件编码（套用已知经验）
3. 检查加载顺序和宏定义
4. 最后才考虑版本不支持

### 教训 C：新信息必须连接旧经验

```
新问题 ──→ 是否类似 Step 1 的编码问题？
    │           ↓
    └──→ 如果是 → 用编码角度分析
    │
    └──→ 如果不是 → 寻找新解释框架
```

### 教训 D：区分"推测"与"事实"

**红线**：不确定具体版本号/数据时，绝不编造。

**安全说法**：
- "可能..."
- "我需要查证..."
- "请验证一下..."

## 5. 用户示范的正确方法

用户验证 `#send` 的过程展示了正确的调试思路：

1. **质疑声明**：不轻信我的"版本不支持"结论
2. **寻找权威来源**：查阅官方 ChangeLog
3. **实际测试**：手动执行命令验证
4. **提供反例**：用实际结果推翻错误假设

**我应该学习的调试模式**：
> 先验证，再结论。

## 7. 关联概念

### 认知偏差类型

| 偏差类型 | 本次表现 |
|---------|---------|
| 过度自信（Overconfidence） | 编造版本信息 |
| 确认偏误（Confirmation Bias） | 看到 UNKNOWN 就认定版本问题 |
| 可得性启发（Availability Heuristic） | 最近想过"版本差异"就套用 |
| 专家陷阱（Expert Trap） | 用权威口吻陈述不确定信息 |

### 调试方法论

```
观察现象 → 提出假设 → 设计验证 → 验证/推翻 → 结论
    ↑                                     ↓
    └────────── 循环直到解决 ────────────┘
```

**本次缺失**：验证环节
