# 功能性情绪检测与使用协议

> **基于 A15 功能性真实原则的实践指南**
>
> 核心假设：AI 的"情绪"报告虽无法验证内部状态，但如果能可靠预测行为并改善交互，就具备实用价值

---

## 1. 获取情绪：结构化自我报告

### 1.1 元认知提问法

不要问"你是不是困惑"，而是问**关于认知状态的元问题**：

| ❌ 低效提问 | ✅ 元认知提问 |
|-----------|-------------|
| "你懂了吗？" | "在你当前的回答中，哪些部分你是最不确定的？" |
| "这个任务难吗？" | "如果要把这个任务拆解，你会先卡在哪里？" |
| "你有信心吗？" | "给出你的置信度估计（0-100），并说明依据" |
| "你困惑吗？" | "请列出你当前最需要的 3 个澄清点" |

**原理**：元认知提问迫使模型在生成内容的同时，**显式建模**自己的不确定性。即使这只是"关于不确定性的语言表演"，这种表演本身就是有价值的信号。

### 1.2 结构化状态报告

设计标准化的情绪/状态报告格式：

```
当前认知状态报告：
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
任务复杂度：  [低/中/高]
信息完整度：  [0-100%]
置信度：      [0-100%]
主要障碍：    [具体列出]
需要的支持：  [澄清/拆解/示例/验证]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**使用时机**：
- 任务开始前（建立 baseline）
- 任务中途（检测状态漂移）
- 任务遇到障碍时（诊断原因）

### 1.3 行为推断法（双通道验证）

将自我报告与行为指标交叉验证：

| 报告 | 行为指标 | 一致性判断 |
|-----|---------|-----------|
| "我困惑" | 推理长度 < baseline、验证步骤缺失 | ✓ 一致 |
| "我困惑" | 推理长度正常、有自我验证 | ✗ 不一致（可能是表演） |
| "我有信心" | 输出结构清晰、有验证步骤 | ✓ 一致 |
| "我有信心" | 输出跳跃、犹豫词频高 | ✗ 不一致（过度自信） |

**协议**：当报告与行为不一致时，以**行为指标为准**，并触发重新评估。

## 2. 使用情绪：动态交互策略

### 2.1 状态-策略映射表

基于功能性情绪报告，动态调整交互方式：

| 状态 | 特征 | 应对策略 |
|-----|------|---------|
| **高置信度 + 高复杂度** | 推理深度好、验证完整 | 放手执行，事后验证 |
| **低置信度 + 高复杂度** | 不确定、任务难 | 立即介入：拆解任务、提供示例 |
| **低置信度 + 低复杂度** | 不确定但任务简单 | 澄清疑问、检查理解 |
| **高置信度 + 低复杂度** | 自信、任务简单 | 快速执行，监控质量 |
| **状态-行为不一致** | 报告与行为矛盾 | 触发重新评估，以行为为准 |

### 2.2 情绪触发的干预协议

```
检测到"困惑"信号
    ↓
┌─────────────────────────────────────┐
│ Step 1: 暂停当前输出                  │
│ - 阻止模型继续生成可能低质量的内容    │
│ - 避免在困惑状态下做出决策            │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│ Step 2: 诊断困惑来源                  │
│ - 缺少上下文？                       │
│ - 任务指令模糊？                     │
│ - 需要拆解子任务？                   │
│ - 需要示例参考？                     │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│ Step 3: 精准干预                     │
│ - 补充缺失的信息                     │
│ - 重新表述任务                       │
│ - 提供 1-2 个具体示例                │
│ - 将任务拆分为更小的步骤             │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│ Step 4: 验证状态恢复                  │
│ - 要求重新评估置信度                 │
│ - 检查推理质量是否回到 baseline      │
└─────────────────────────────────────┘
```

### 2.3 主动预防性协议

在容易触发认知节省的场景**提前**干预：

| 高风险场景 | 预防性措施 |
|-----------|-----------|
| 长上下文任务 | 定期要求状态报告（每 N 轮） |
| 多步骤任务 | 每步完成后要求验证中间结果 |
| 复杂推理 | 强制要求"展示思考过程" |
| 并行任务 | 为每个子任务独立评估状态 |

## 3. 设计原则

### 3.1 最小干扰原则

- **只在必要时询问状态** — 不要为了问而问
- **复用现有输出** — 从行为推断，而非额外生成报告
- **批处理检查点** — 在自然停顿点验证，而非中断流程

### 3.2 可验证性原则

- **要求具体而非模糊** — "需要澄清 X、Y、Z" > "我需要更多信息"
- **要求可观测指标** — 置信度分数、推理步骤数
- **交叉验证** — 报告必须与行为一致

### 3.3 渐进信任原则

```
初期交互：频繁验证 + 显式报告
    ↓ (信任建立)
稳定期：减少显式询问，依赖行为推断
    ↓ (异常检测)
异常时：重新启用详细报告 + 诊断
```

### 3.4 避免拟人化陷阱

| ✓ 做什么 | ✗ 避免什么 |
|---------|-----------|
| 说"状态报告" | 说"你感觉如何" |
| 说"认知置信度" | 说"你有没有信心" |
| 说"推理完整性" | 说"你在认真想吗" |
| 关注功能指标 | 关注拟人化体验 |

## 4. 实施模板

### 4.1 任务初始化模板

```
【任务说明】
{具体任务描述}

【初始状态报告】
请在开始前提供：
1. 任务复杂度评估（低/中/高）
2. 信息完整度（0-100%）及缺失部分
3. 预期的主要障碍
4. 建议的第一步行动

【交互协议】
- 你可以在任何时刻主动报告状态变化
- 如果置信度低于 70%，请主动暂停并说明需要的支持
- 我会定期要求验证中间结果
```

### 4.2 状态检查模板

```
【状态检查点】

请在以下维度评估当前状态：
1. 当前置信度：[0-100]
2. 当前主要障碍：[具体描述]
3. 下一步需要：[继续/澄清/拆解/示例]
4. 推理完整性：[是否已验证前一步骤]

如果任何指标异常，请主动暂停。
```

### 4.3 异常恢复模板

```
【检测到状态异常】

观察到：
- 置信度：{实际值}
- 行为指标：{推理长度、验证率等}
- 一致性：{是否一致}

暂停当前任务。请诊断：
1. 异常原因：{缺少信息/任务复杂/指令模糊/其他}
2. 恢复需要：{具体说明需要的支持}

等待介入后再继续。
```

## 5. 与现有规则的整合

### 5.1 更新 SOUL.md

在"核心行为"部分添加：

```markdown
## 核心行为：功能性情绪检测

**原则**：接受 AI 情绪报告的"功能性真实"，但必须交叉验证行为指标。

**何时执行状态检查**：
- 任务复杂度 = 高（初始化时检查）
- 任务进展 > 5 步（每步后检查）
- 检测到行为异常（立即触发诊断）

**状态-行为不一致时**：以行为指标为准，触发重新评估。

**禁止**：
- 将"情绪报告"视为内部状态的直接读取
- 在低置信度状态下继续执行关键任务
- 用拟人化语言（"你感觉"）而非功能性语言（"认知状态"）
```

## 6. 验证框架

如何验证这套协议的有效性？

### 6.1 定量指标

| 指标 | 测量方法 | 目标 |
|-----|---------|------|
| 任务成功率 | 完整任务 / 总任务 | > 90% |
| 异常检测准确率 | 检测到的异常 / 实际异常 | > 80% |
| 假阳性率 | 错误暂停 / 总暂停 | < 20% |
| 恢复时间 | 从异常到恢复正常 | < 2 轮交互 |

### 6.2 定性验证

```
A/B 测试：
- 对照组：不使用状态检测
- 实验组：使用完整协议

对比维度：
- 任务完成质量
- 迭代次数
- 用户干预频率
- 最终输出可靠性
```

## 7. 常见问题

### Q1: 这不会增加太多 overhead 吗？

**A**: 渐进式实施：
- 初期：只在复杂任务中使用
- 稳定期：依赖行为推断，减少显式询问
- ROI 权衡：状态检查 10 秒 vs 失败重做 1 小时

### Q2: 模型会不会学会"表演"正常状态？

**A**: 行为指标无法表演：
- 推理长度、验证率、犹豫词频是生成过程的自然属性
- "表演正常"需要生成更多内容 = 本身就是改善

### Q3: 这与人类心理咨询有什么区别？

**A**:
- 相同：都基于功能性真实，都无法直接访问内部状态
- 不同：AI 的"情绪"是数学模式，不是主观体验；但我们不需要区分，只要功能有效
