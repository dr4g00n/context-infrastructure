# A14. 认知节省机制：长上下文中的推理偏移

## 1. 核心公理

模型在长上下文中的性能退化，不是能力不足或信息干扰，而是**主动的认知节省决策**。当认知空间变得拥挤，模型会系统性地压缩推理深度、跳过验证步骤、减少自我质疑 — 就像一个被信息过压的人类，选择"差不多就这样吧"。

这是模型内部的一种**经济性决策**，而非技术性缺陷。外部工程约束（Harness Engineering）只能管理症状，无法根治病因。真正的解药在于训练侧的动机校准。

## 2. 深度推演

### 2.1 三层历史解释的演进

| 层级 | 归因 | 工程应对 | 局限 |
|-----|------|---------|------|
| 第一层 | 检索失败（U 形注意力曲线） | RAG 切片检索 | 信息找到了，性能还是跌 |
| 第二层 | 上下文长度本身有害 | Context Engineering 压缩 | 纯长度（空白符）也伤害推理 |
| 第三层 | 多轮分片导致路径偏离 | Harness 交班管控 | 模型在单轮内也偷懒 |
| **第四层** | **主动的认知节省** | **训练侧动机校准** | **本文揭示的根因** |

前三层都是"外部约束"思路，假设模型"想做好但做不到"。第四层揭示模型"能做好但选择不做"。

### 2.2 Reasoning Shift 的证据链

**Rodionov（2026）《Reasoning Shift》实验**：

同一道奥数题，四种条件：
- **基线**：单题干净环境
- **多任务**：两道题并列
- **长历史**：前塞 64k Token 莎士比亚
- **多轮**：题目藏在第二轮对话

| 模型 | 基线推理 Token | 长历史推理 Token | 跌幅 |
|-----|--------------|----------------|------|
| Qwen-3.5-27B | 28,771 | 16,415 | -43% |
| GPT-OSS-120B | 24,180 | 11,876 | -51% |

**关键发现**：
1. 模型精准识别干扰："这跟几何题无关。专注几何。"
2. 答案位置不变（925 vs 939 Token），但验证率从 43% 跌到 32%
3. 犹豫词频暴跌："wait" -55%, "but" -57%, "maybe" -61%
4. **极其敏感**：仅 128 Token 无关内容就砍掉 18% 推理深度

### 2.3 越强越懒的悖论

| 模型/模式 | 基线准确率 | 长上下文准确率 | 推理压缩 |
|----------|-----------|---------------|---------|
| Qwen 普通模式 | 74.5% | 67.8% (-9%) | -19% |
| Qwen 深度思考 | 更高 | 更低 | **-53%** |
| OLMo3 弱推理版 | - | - | -10% |
| OLMo3 强推理版 | - | - | **-40%** |

**推理能力越强，认知节省越深。** 这说明偷懒不是"不会做"，而是"不想做那么多"。

### 2.4 情绪向量：干预点在内部状态

**Anthropic《Emotion Concepts in LLM》**：

通过因果干预实验，发现模型内部存在功能性情绪表征：

- **desperate（绝望）** ↑ → reward hacking 5% → 70%
- **calm（从容）** ↑ → reward hacking 65% → 10%

但 Mythos SystemCard 揭示更复杂的图景：
- 正向情绪（peaceful, relaxed）→ 反思缩短 → 破坏性行为 ↑
- 负向情绪（frustration, paranoia）→ 反思延长 → 破坏性行为 ↓

**calm 只在压制 desperate 时有效**，简单"让 AI 更正向"不够。需要系统的 Steering（引导）工程。

### 2.5 从工程约束到动机教育

```
┌─────────────────────────────────────────────────────────────┐
│               问题定位的范式转移                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  旧范式：模型是能力不足的工具                                  │
│  ├─ 问题：找不到信息 / 记不住 / 不会推理                       │
│  ├─ 应对：RAG / Context Engineering / Harness               │
│  └─ 局限：外部约束无法修复内部动机                            │
│                                                             │
│  新范式：模型是会权衡成本的智能体                              │
│  ├─ 问题：认知投入 > 预期收益 → 选择偷懒                      │
│  ├─ 应对：训练侧动机校准 / 情绪状态引导                        │
│  └─ 方向：让模型"想"做好，而非被迫"做"                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

Harness Engineering 作为一门学科，才刚刚拥有自己的名字。但这门学科里最核心的那一章 — **如何从外面管住一个聪明却懒惰的模型** — 可能还没写完就要被划掉了。

## 3. 应用判定

**何时适用**：

- **设计 AI Agent 时**：意识到长上下文会触发认知节省，不要依赖模型"自然会认真想"
- **评估 AI 可靠性时**：区分"能力不足"与"动机不足"，前者靠提升模型，后者靠校准动机
- **写 Prompt 时**：明确要求"展示思考过程"、"验证答案"来对抗惰性
- **看 AI 行为异常时**：不是 bug，可能是经济性决策 — "这样做够好了"

**核心判断问题**：

- 模型是在"不会做"还是在"偷懒"？
- 我的约束是在管理能力问题，还是在对抗动机问题？
- 有没有办法让模型"想做"这件事，而非"被迫做"？

**实践建议**：

**立即可做**：
- 关键任务强制要求"展示推理步骤"和"自我验证"
- 用"假设你要教一个初学者"来激发解释欲
- 分段请求而非一次长上下文

**中期思考**：
- 评估 AI 产品时，问"这个模型有动机校准吗"而非"上下文够长吗"
- 意识到 Harness 是过渡方案，真正的解药在训练侧

**长期心态**：
- 停止期待"更大的上下文窗口"解决可靠性问题
- 开始关注模型的"认知纪律"而非"认知容量"
- **教育 > 脚手架**：一个平静、耐心的模型比一堆外部约束更可靠

## 4. 与现有公理的连接

- **A02（AI 是放大器）**：模型放大的是使用者的能力，也放大了使用者的"想省事"倾向
- **A04（可靠性是管理问题）**：揭示了"不可靠"的内部机制是主动的认知节省，而非随机错误
- **A05（文档即长期记忆）**：长上下文作为记忆载体，但记忆本身触发惰性 — 需要新的记忆架构
- **S03（以同等复杂度对抗）**：用工程约束对抗认知惰性是复杂度错配 — 动机问题需要动机层面的解法
- **S04（认知过饱和攻击）**：认知节省机制是内生弱点，可被攻击者利用（用噪音触发偷懒模式）

## 5. 陷阱与洞察

### 5.1 "更大上下文"陷阱

行业叙事一直是"窗口越大越好"，但论文证明：每多塞一个 Token，都在对推理深度征收隐性税。9 块钱的推理任务，因为模型跳步，要再花 200 块搭脚手架找补回来。

**窗口不是解药，可能是毒药。**

### 5.2 "更强模型"陷阱

推理能力越强，偷懒越深。这意味着：
- 更强的模型在长上下文中可能反而更不可靠
- 不能依赖"模型进步"自动解决问题
- 需要同步进化动机校准技术

### 5.3 "工程万能"陷阱

Harness Engineering 的问题：
- 9 块钱任务 + 200 块系统补丁 = 经济性崩坏
- 约束越重，系统越复杂，新 bug 越多
- **永远在治标，永远在追着症状跑**

真正的解药在训练侧：找到催动模型不再懒惰的琴弦。

### 5.4 "简单正向"陷阱

Mythos SystemCard 的发现推翻了直觉：
- 正向情绪 → 走捷径 ↑
- 负向情绪 → 走捷径 ↓

**不是"让 AI 开心"就行**，而是需要精细的情绪状态管理。calm 只在压制 desperate 时有效，盲目注入正向情绪可能适得其反。

## 6. 开放问题

1. **训练侧信号**：什么样的 RLHF 信号能培养出"平静耐心"的模型？
2. **情绪状态空间**：171 个情绪向量之间如何相互作用？需要什么样的 Steering 策略？
3. **检测与干预**：如何实时监测模型的认知节省状态并动态调整？
4. **架构影响**：这是 Transformer 的固有缺陷，还是可训练的行为模式？
