# S01. 安全是关系属性，非对象属性

## 1. 核心公理

安全不是数据、系统或资产本身的属性，而是"人-对象关系"的属性。数据本身无安全属性，只有当人的意图介入时，安全问题才产生。因此，评估安全性的第一步不是"这个对象是否安全"，而是"谁以什么意图接触这个对象"。

## 2. 深度推演

### 2.1 统一视角重新理解安全技术

这个公理提供了一个统一视角来理解所有安全技术——它们本质上都在做同一件事：隔离人与数据之间的关系。

| 技术 | 表面作用 | 本质：隔离什么 |
|------|---------|--------------|
| 加密 | 保护数据 | 隔离"理解"：人看不懂数据 |
| 访问控制 | 限制访问 | 隔离"接触"：人无法触及数据 |
| 网络隔离 | 划分边界 | 隔离"路径"：人无法到达数据 |
| 审计日志 | 记录行为 | 隔离"隐形"：人无法隐藏行为 |

由此推论：
- **漏洞** = 隔离失效的地方
- **攻击** = 突破隔离的尝试
- **防御** = 加强隔离的强度
- **零信任** = 持续验证隔离的有效性

### 2.2 安全设计的第一步：识别关系，而非识别对象

传统安全设计从"这个对象有什么风险"开始，是以对象为中心的视角。关系视角要求先回答三个问题：

1. **谁**会和这个对象发生关系？（主体识别）
2. 以**什么意图**发生关系？（动机建模）
3. **如何**发生关系？（路径分析）

只有在这三个维度清晰之后，隔离机制的设计才有意义。

### 2.3 隔离的代价与平衡点

过度隔离 = 不信任任何人 = 系统效率下降。安全设计的真实目标是：**在可接受风险下，最小化隔离成本**。

| 隔离强度 | 安全性 | 效率 | 适用场景 |
|---------|--------|------|---------|
| 无隔离 | 低 | 高 | 内部信任网络 |
| 基础隔离 | 中 | 中 | 通用业务系统 |
| 严格隔离 | 高 | 低 | 高价值/高风险数据 |
| 完全隔离 | 最高 | 最低 | 核心机密 |

### 2.4 跨域迁移

这个公理不止适用于数据安全。**任何涉及"保护某个对象免受某类主体影响"的设计问题，都可以用关系视角重新框架**：

- **产品权限设计**：不问"这个功能是否安全"，而问"哪类用户以什么动机会使用这个功能"
- **组织信息流动**：不问"这个信息是否机密"，而问"谁需要知道、谁不应该知道、为什么"
- **个人隐私边界**：不问"这件事能不能说"，而问"说给谁、在什么情境下"

## 3. 应用判定

**何时适用**：

- 评估任何系统、产品或信息的安全性时
- 设计访问控制、权限体系时
- 发现安全措施"感觉过重"或"感觉不足"时——通常是关系模型不清晰
- 讨论"这个数据/功能需不需要保护"时

**触发信号**：
- 有人说"这个数据很敏感"→ 反问：对谁敏感？在什么情境下？
- 有人说"我们需要加密这个"→ 反问：你在隔离谁与数据之间的什么关系？

## 4. 陷阱与洞察

### 4.1 "数据分级"陷阱

安全等级不是数据的固有属性。同一份数据，对内部员工是公开的，对竞争对手是机密的，对监管机构又是必须公开的。关系属性视角要求把"谁"和"情境"纳入分级逻辑，而非给数据贴一个脱离情境的标签。

### 4.2 "技术能解决一切"陷阱

安全技术再强，也无法替代对关系的理解。如果不清楚谁会以什么动机接触某个系统，再好的加密和访问控制也可能保护了错误的对象，或在正确的地方保护不足。技术是实现隔离的手段，关系模型是确定隔离目标的前提。

## 5. 相关公理

- **S02: 防御目标是改变成本结构** — S01 确定"隔离谁"，S02 确定"以什么方式隔离更有效"
- **V02: 可验证性是信任的地基** — 隔离机制需要可验证，否则无法知道隔离是否失效
- **T03: 上下文隔离是多 Agent 价值** — 信息隔离原则在多 Agent 系统中的具体应用

## 6. 实践建议

**立即可做的事**：

1. 拿出当前正在设计的系统，列出所有"会和系统发生关系的主体"——不只是"用户"，还包括运维、开发、自动化脚本、第三方服务
2. 对每个主体，建立意图矩阵（正常意图 / 可能误操作 / 潜在恶意意图）
3. 检查现有安全措施：它们在隔离哪段关系？有没有遗漏的关系？有没有过度隔离？

**长期的心态转变**：

- 停止问"这个东西安不安全"，开始问"谁以什么意图接触这个东西"
- 停止把安全等级当作对象的固有属性，开始把它当作关系的函数
- 停止用技术来覆盖关系模型不清晰的问题，开始先把关系搞清楚再选技术
